# Lua-Authored Shader Editor Schema Plan

Status: Proposed implementation plan
Date: 2026-03-10
Owners: dashboard + rendering + scripting + runtime metadata

Primary references:
- `docs/features/shader_editor_dashboard_implementation_spec.md`
- `docs/plans/shader_editor_dashboard_plan.md`
- `docs/decisions/dr-0029_runtime_shader_family_taxonomy_and_lua_authoring_model.md`
- `docs/component_authoring_guide.md`
- `data/scripts/assets/registry.lua`
- `crates/sidereal-game/src/generated/components.rs`

## 1. Objective

Define a canonical shader editor schema model that supports:

1. `min` / `max` / `step`
2. enum-backed dropdowns
3. grouped presets
4. labels, descriptions, and UI hints
5. dashboard consumption from generated runtime metadata rather than dashboard-only parsing

The accepted direction is:

1. WGSL remains the source of truth for shader ABI and actual uniform declarations.
2. Lua becomes the canonical authoring surface for advanced editor metadata and presets.
3. The final merged shader schema must be generated by Rust and exposed through the existing `GeneratedComponentRegistry` resource rather than a new dashboard-only metadata path.

## 2. Why This Direction

This matches the repo's existing content-authoring contract:

1. `dr-0029` already says Lua owns shader asset IDs, parameter payloads, texture bindings, content presets, and effect spawn templates.
2. The dashboard shader editor spec already uses Lua asset metadata for dependencies and roles.
3. The component authoring guide already establishes `GeneratedComponentRegistry` as the authoritative generated metadata surface for dashboard/editor behavior.

The main problem today is not that shader schema is impossible. The problem is that it is split across the wrong layers:

1. the dashboard infers preview uniforms ad hoc from WGSL text,
2. advanced control metadata does not exist in a canonical source,
3. no generated runtime resource currently carries shader editor metadata,
4. presets would currently have to live in frontend-only maps or handwritten editors.

That is exactly the kind of metadata drift the generated resource model is meant to avoid.

## 3. Non-Negotiable Decisions

### 3.1 Canonical authoring source

Canonical advanced shader editor metadata must live in Lua asset authoring, not in dashboard-only TypeScript maps.

Accepted source:

1. `data/scripts/assets/registry.lua`

Rejected as canonical source:

1. dashboard-local schema tables,
2. handwritten per-shader React mappings,
3. freeform WGSL comments as the primary metadata source.

### 3.2 Existing resource requirement

The merged shader editor schema must be generated into the existing `GeneratedComponentRegistry` resource.

Direction:

1. extend the resource shape rather than introducing a second independent BRP resource just for shaders,
2. preserve existing component entries and semantics,
3. add shader entries as an adjacent generated metadata section.

### 3.3 Authority split

The ownership split must remain explicit:

1. WGSL owns real uniforms, bindings, and type shape.
2. Lua owns editor presentation metadata and presets.
3. Rust owns parsing, validation, merge rules, and generated resource output.
4. Dashboard consumes the generated result and does not redefine schema policy.

## 4. Proposed Resource Model

## 4.1 Extend `GeneratedComponentRegistry`

Current shape:

1. `entries: Vec<ComponentRegistryEntry>`

Proposed shape:

1. `entries: Vec<ComponentRegistryEntry>`
2. `shader_entries: Vec<ShaderEditorRegistryEntry>`

Do not rename the resource in the first pass. The name is slightly component-centric, but preserving the existing resource path keeps the dashboard/BRP integration stable while the resource grows into the broader editor metadata registry it is already functionally becoming.

If the name becomes too misleading later, rename only in a follow-up change after dashboard and BRP readers already consume the expanded shape.

## 4.2 Proposed Rust types

Representative direction:

```rust
pub struct ShaderEditorRegistryEntry {
    pub asset_id: String,
    pub source_path: String,
    pub shader_family: Option<String>,
    pub shader_role: Option<String>,
    pub dependencies: Vec<String>,
    pub entry_points: Vec<String>,
    pub bind_layout_schema: Vec<ShaderBindingSchema>,
    pub uniform_schema: Vec<ShaderUniformFieldSchema>,
    pub presets: Vec<ShaderEditorPreset>,
}

pub struct ShaderUniformFieldSchema {
    pub field_path: String,
    pub binding_name: String,
    pub display_name: String,
    pub value_kind: ComponentEditorValueKind,
    pub min: Option<f64>,
    pub max: Option<f64>,
    pub step: Option<f64>,
    pub options: Vec<ShaderEditorOption>,
    pub description: Option<String>,
    pub group: Option<String>,
    pub default_value: Option<serde_json::Value>,
}

pub struct ShaderEditorPreset {
    pub preset_id: String,
    pub display_name: String,
    pub description: Option<String>,
    pub values: serde_json::Value,
}
```

Important constraint:

1. reuse existing editor-schema value kinds where possible so dashboard renderers do not need two unrelated widget taxonomies.

## 5. Lua Authoring Model

## 5.1 Placement

Shader editor metadata lives alongside the shader asset definition in `data/scripts/assets/registry.lua`.

Example direction:

```lua
{
  asset_id = "planet_visual_wgsl",
  shader_family = "world_polygon_planet",
  source_path = "shaders/planet_visual.wgsl",
  content_type = "text/plain; charset=utf-8",
  dependencies = {},
  bootstrap_required = true,
  editor_schema = {
    uniforms = {
      atmosphere_alpha = {
        label = "Atmosphere Alpha",
        kind = "float",
        min = 0.0,
        max = 1.0,
        step = 0.01,
        default = 0.48,
        description = "Alpha used by the atmosphere rim pass.",
        group = "Atmosphere",
      },
      blend_mode = {
        label = "Blend Mode",
        kind = "enum",
        default = "screen",
        options = {
          { value = "screen", label = "Screen" },
          { value = "add", label = "Add" },
          { value = "lighten", label = "Lighten" },
        },
      },
    },
    presets = {
      {
        preset_id = "earth_like",
        label = "Earth-like",
        values = {
          atmosphere_alpha = 0.48,
          city_lights = 0.04,
        },
      },
      {
        preset_id = "dry_world",
        label = "Dry World",
        values = {
          atmosphere_alpha = 0.18,
          city_lights = 0.0,
        },
      },
    },
  },
}
```

## 5.2 Schema semantics

Lua metadata is not allowed to invent uniforms that do not exist in WGSL.

Lua may define:

1. labels
2. descriptions
3. groups/sections
4. `min` / `max` / `step`
5. enum options
6. presets
7. default values
8. optional UI hints such as `widget = "slider"` where later needed

Lua may not define:

1. arbitrary fake uniforms absent from WGSL,
2. bind group layout that conflicts with actual shader declarations,
3. incompatible type overrides that contradict the parsed WGSL type.

## 5.3 Enums

Enums need explicit policy because WGSL itself does not have a native editor-facing enum type for uniforms.

Accepted direction:

1. enum uniforms are represented in WGSL as scalar numeric uniforms,
2. Lua supplies the allowed options and labels,
3. Rust validates that option values are representable for the underlying numeric type,
4. dashboard renders a dropdown and writes the selected numeric payload back into the uniform value.

## 5.4 Presets

Presets are content-level authoring data and therefore belong in Lua.

Preset rules:

1. each preset must target one shader asset entry,
2. preset values must only reference declared uniforms,
3. preset values must pass type validation against the merged schema,
4. the dashboard may apply a preset to preview state and later to exported component defaults.

## 6. WGSL Role

WGSL remains authoritative for:

1. uniform existence,
2. scalar/vector/struct shape,
3. entry points,
4. bind group and binding indices,
5. texture/sampler declarations.

The merge pipeline must derive these directly from parsed WGSL rather than trusting Lua to restate them.

## 6.1 Comments policy

WGSL comments may be supported later as a non-authoritative hint layer only.

Allowed future uses:

1. field descriptions,
2. display labels,
3. quick preview notes for artists.

Not allowed as the primary source of truth:

1. ranges,
2. enums,
3. presets,
4. authoritative UI contracts.

Reason:

1. comments are too easy to drift,
2. they are awkward for large presets,
3. they force editor metadata into shader source instead of keeping content authoring in Lua.

## 7. Generation Pipeline

## 7.1 Rust generation flow

The resource generation pipeline should become:

1. load/parse Lua asset registry,
2. select shader assets from the registry,
3. read canonical WGSL source for each shader asset,
4. parse WGSL with a real Rust parser/reflection layer,
5. derive base binding and uniform schema from WGSL,
6. merge Lua `editor_schema` metadata onto the derived fields,
7. validate the merged result,
8. publish the final `shader_entries` inside `GeneratedComponentRegistry`.

## 7.2 Parser choice

The implementation should use a Rust parser/reflection path, not the current dashboard regex parser, for the generated resource.

Preferred direction:

1. promote reusable WGSL reflection helpers into a shared Rust crate or module,
2. use Naga-based parsing as the canonical implementation path,
3. keep the current frontend regex extraction only as a temporary fallback until the generated resource is wired through.

## 7.3 Merge rules

Merge behavior must be deterministic:

1. WGSL creates the base field list.
2. Lua may override presentation metadata for matching field names.
3. Lua-only fields that do not match parsed uniforms are validation errors.
4. Type mismatches between Lua metadata and WGSL shape are validation errors.
5. Missing Lua metadata is allowed; Rust supplies conservative defaults.

## 7.4 Validation errors

Generation should fail loudly for:

1. duplicate preset IDs within one shader entry,
2. duplicate uniform metadata keys,
3. enum metadata on non-numeric uniforms unless explicitly supported,
4. preset values for unknown uniforms,
5. `min > max`,
6. invalid default values,
7. Lua schema references to missing shader source files.

## 8. Dashboard Changes

## 8.1 Main contract

The dashboard must stop treating shader editor schema as a frontend-inferred-only contract once the generated resource exists.

Dashboard direction:

1. read shader schema from the generated BRP resource,
2. use the same widget mapping layer already used by generic component editors where possible,
3. preserve temporary fallback inference only while generated shader entries are absent.

## 8.2 Data access path

Current dashboard behavior:

1. shader metadata comes from `/api/shaders`,
2. preview uniforms come from local WGSL parsing in TypeScript.

Planned behavior:

1. shader asset listing still comes from `/api/shaders` for file catalog/navigation concerns,
2. schema/preset metadata is resolved from `GeneratedComponentRegistry.shader_entries`,
3. shader workbench merges file-load state with generated shader metadata by `asset_id` or canonical `source_path`.

This keeps dashboard navigation/server file access separate from authoritative editor metadata generation.

## 8.3 UI features to add

Dashboard schema support should include:

1. slider rendering when `min` and `max` exist,
2. numeric input precision driven by `step`,
3. enum dropdowns for schema fields with `options`,
4. grouped sections based on schema `group`,
5. preset picker with one-click apply,
6. reset-to-schema-default action,
7. display of descriptions/help text.

## 8.4 Optimistic behavior

Dashboard optimistic state should continue to operate locally, but the source of widget shape and validation hints must come from the generated shader schema rather than local heuristics.

## 8.5 Shader settings export workflow

The main dashboard component editor must support the existing shader-settings tuning workflow where an operator edits a live component and then copies a canonical JSON payload back into the Rust default constant.

This is specifically required for components such as:

1. `PlanetBodyShaderSettings`
2. `SpaceBackgroundShaderSettings`
3. `StarfieldShaderSettings`
4. `ThrusterPlumeShaderSettings`
5. other shader-settings-style gameplay components that currently use `DEFAULT_*_SHADER_SETTINGS_JSON`

Accepted UX:

1. add `Copy Default JSON` to the component editor for shader settings components,
2. optionally add `Copy Rust Const` as a convenience wrapper later,
3. keep the export shape identical to the JSON consumed by the Rust default constants rather than inventing a dashboard-only serialization format.

Export rules:

1. export from the currently edited component payload,
2. serialize `Vec2` / `Vec3` / `Vec4` in the same JSON shape expected by the Rust component default constants,
3. round floats to stable schema-driven precision so floating-point noise is not copied into defaults,
4. preserve deterministic field ordering, preferably schema order,
5. exclude dashboard-only metadata and transient editor state.

Relationship to Lua shader schema work:

1. this export flow is for gameplay component defaults,
2. it is not a replacement for Lua-authored shader presets,
3. later the dashboard may also support `Copy Lua Preset`, but that must remain a separate output path from `Copy Default JSON`.

## 9. Migration Plan

## 9.1 Phase 1: Resource shape extension

1. Extend `GeneratedComponentRegistry` with `shader_entries`.
2. Register reflection/serde types needed for shader schema records.
3. Keep existing `entries` behavior unchanged.
4. Add resource parsing support in dashboard TypeScript.

Exit criteria:

1. BRP can inspect shader entries on the resource,
2. existing component editor flows remain unchanged.

## 9.2 Phase 2: Lua schema authoring support

1. Extend Lua asset registry parsing to accept optional `editor_schema` blocks for shader assets.
2. Validate and deserialize the Lua metadata into Rust-owned schema types.
3. Keep empty/missing schema blocks valid.

Exit criteria:

1. one or two real shader assets can declare ranges and presets in Lua,
2. parser rejects malformed schema consistently.

## 9.3 Phase 3: WGSL-derived schema generation

1. Add Rust-side WGSL reflection for uniform/binding extraction.
2. Merge the derived base schema with Lua metadata.
3. Publish merged output into `GeneratedComponentRegistry.shader_entries`.

Exit criteria:

1. generated shader entries include real uniform fields,
2. dashboard no longer needs regex inference for those covered shaders.

## 9.4 Phase 4: Dashboard consumption

1. Update dashboard shader workbench to prefer generated shader schema.
2. Add enum and preset UI controls.
3. Add schema-aware `Copy Default JSON` export support for shader settings components in the main dashboard editor.
4. Keep a fallback path only for shaders not yet present in the generated resource.

Exit criteria:

1. shader inspector controls are driven by generated metadata,
2. presets can be applied without dashboard-only configuration,
3. shader settings edited in the main dashboard can be copied back into Rust `DEFAULT_*_SHADER_SETTINGS_JSON` constants without manual cleanup.

## 9.5 Phase 5: Cleanup

1. Remove obsolete dashboard-only per-shader schema logic where superseded.
2. Reduce the frontend regex parser to fallback/debug use or remove it entirely if coverage is complete.
3. Update docs to state the final source-of-truth policy clearly.

## 10. Testing Plan

Required Rust coverage:

1. Lua schema parsing tests,
2. WGSL reflection tests for scalar/vector/struct uniforms,
3. merge tests for valid and invalid metadata,
4. generated resource tests covering `shader_entries`,
5. roundtrip serialization tests for BRP-visible resource payloads.

Required dashboard coverage:

1. parsing/reading of `shader_entries` from the resource,
2. enum control rendering,
3. preset apply behavior,
4. `Copy Default JSON` export for shader settings components,
5. schema-driven float normalization in exported JSON,
6. fallback behavior when generated schema is absent,
7. schema-driven precision/range UI tests.

## 12. Open Questions

1. Whether the existing `GeneratedComponentRegistry` name should be preserved long-term or later renamed to something broader such as `GeneratedEditorRegistry`.
2. Whether shader presets should remain shader-workbench-only initially or also become reusable for gameplay component default export/import flows.
3. Whether descriptions/help text should support a small WGSL comment fallback when Lua metadata is missing.
4. Whether schema generation should happen in one shared crate consumed by both runtime and gateway/dashboard helpers, or only in one runtime crate with serialized exposure outward.

## 13. Recommended First Slice

The highest-leverage first slice is:

1. extend `GeneratedComponentRegistry` with `shader_entries`,
2. add Lua-side `editor_schema` parsing for one shader asset,
3. add Rust-side WGSL uniform extraction for scalar/vector uniforms only,
4. merge Lua `min` / `max` / `step` and one preset,
5. update the dashboard shader workbench to read and render that schema from the resource,
6. add `Copy Default JSON` for one existing shader-settings component to prove the export path and precision normalization rules.

That slice proves the architecture without needing full shader-family coverage on day one.
